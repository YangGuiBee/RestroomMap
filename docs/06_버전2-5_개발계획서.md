# 화장실SOS(RestroomMap) 버전2~5 개발계획서

| 항목 | 내용 |
|---|---|
| 문서버전 | v1.0 |
| 작성일 | 2026-07-17 |
| 전제 | 버전1(공공화장실, 백엔드 없음)이 실배포 완료된 상태 기준 |

버전1에서 이미 스키마·구조 단계에 예약해 둔 확장 포인트(`source`, `category` 필드, i18n 구조, 지도 레이어 추상화)를 실제로 사용하는 단계다. 버전이 올라갈수록 아키텍처 변경 폭이 커지므로, **버전2에서 백엔드를 도입하는 순간이 이 프로젝트의 가장 큰 분기점**이다.

---

## 버전2 — 민간 제보 + 관리자 대시보드

### 목표
사용자가 직접 화장실 정보를 제보(신규 등록/오류 신고)할 수 있게 하고, 운영자가 이를 검수·반영할 수 있는 관리 도구를 제공한다.

### 주요 기능
| 기능 | 설명 |
|---|---|
| 화장실 제보 | "이 근처 화장실 제보" 버튼(버전1 UI에 자리만 확보됨) → 위치·이름·시설정보 입력 폼 |
| 정보 오류 신고 | 상세시트의 "정보 오류 신고" 버튼(i18n 문자열 이미 존재) 활성화 |
| 관리자 대시보드 | 로그인 필요. 제보 검수(승인/반려), 공공데이터 재수집 트리거, 사용 현황(조회수/안내시작 클릭 등 집계) 확인 |
| 데이터 갱신 자동화 | GitHub Actions로 월 1회 자동 재수집(현재는 수동) 검토 |

### 아키텍처 변경 (버전1 대비 가장 큰 변화)
- **백엔드 도입 필요**: 제보 데이터 저장·검수 상태 관리, 관리자 인증에는 정적 호스팅만으로 불가능. Vercel Serverless Functions 또는 별도 경량 백엔드(Supabase 등 BaaS) 검토
- DB 필요: 제보 원본, 검수 상태, 관리자 계정을 저장할 최소한의 RDB/문서DB
- 데이터 스키마: `source: "user"` 필드를 실제로 사용(버전1엔 예약만 되어 있고 전부 `"public"`)
- 프라이버시 원칙 유지: 제보자의 위치정보는 여전히 서버에 상시 수집·저장하지 않음(제보 시점의 좌표만 1회성으로 받음)

### 선결 과제
- 카카오 로컬 API 이용약관상 좌표 재공급 관련 확인(제안서 작성 중 발견한 이슈, 민간 제보 좌표는 자체 소유이므로 직접 영향은 적으나 공공데이터 좌표와 혼재 시 검토 필요)
- 관리자 인증 방식 결정(간단한 비밀번호 vs OAuth)

## 버전3 — 지하철 화장실

### 목표
지하철역 내 화장실(대합실 내부, 개찰구 안쪽 등 공공화장실과 접근 경로가 다른 시설)을 추가 커버리지로 편입.

### 주요 기능
| 기능 | 설명 |
|---|---|
| 지하철 화장실 데이터 추가 | 서울교통공사 등 지하철 운영기관 공공데이터 연동 |
| 카테고리 필터 | 지도/카드에서 "공공화장실만/지하철만/전체" 필터 |
| 접근 경로 안내 | 개찰구 안/밖 여부 등 지하철 특유 정보 표시(상세시트 필드 확장) |

### 아키텍처 변경
- 데이터 스키마: `category: "subway"` 필드 실사용(버전1엔 예약만)
- 지하철역은 공공화장실보다 좌표 밀집도가 낮아(역 단위) 버전1에서 겪은 "픽셀 겹침" 이슈가 상대적으로 적을 것으로 예상되나, 환승역처럼 여러 출입구가 밀집된 경우 동일 이슈 재점검 필요
- 데이터 갱신 주기를 공공화장실과 다르게 관리해야 할 수 있음(운영기관이 다르므로)

## 버전4 — 다국어 지원

### 목표
외국인 관광객·거주자도 사용할 수 있도록 다국어 UI 제공.

### 주요 기능
| 기능 | 설명 |
|---|---|
| 언어 선택 | 영어/일본어/중국어 등 추가(수요 기반 우선순위 결정) |
| 자동 언어 감지 | 브라우저 `navigator.language` 기반 초기 언어 자동 선택 |

### 아키텍처 변경
- **버전1에서 이미 대비되어 있음**: 모든 UI 문자열이 `src/i18n/ko.json` 단일 파일로 관리되므로, 언어별 JSON 파일(`en.json`, `ja.json` 등) 추가 + 간단한 언어 선택 상태만 추가하면 됨
- 개방시간 파서(`src/lib/openHours.ts`)의 요일 판별 로직은 한국어 텍스트("휴무", "월요일" 등) 정규식에 의존 — 원본 데이터 자체가 한국어이므로 이 부분은 다국어화 대상이 아니라 **번역이 아닌 그대로 유지**(파싱 결과인 open/closed 상태만 다국어 라벨로 표시)
- 시설명·주소는 번역하지 않고 원문(한국어) 그대로 노출하는 것이 현실적(정확성 문제)

## 버전5 — 해외 확장

### 목표
한국 외 지역(예: 일본, 동남아 관광지)으로 서비스 확장.

### 주요 기능
| 기능 | 설명 |
|---|---|
| 지역별 데이터 소스 연동 | 국가별 공공화장실 오픈데이터(있는 국가 한정) 또는 크라우드소싱(버전2 제보 기능 재활용) |
| 지도 SDK 교체 | 카카오맵은 해외 지도 품질이 낮으므로 OSM/Leaflet 등으로 전환 |
| 내비 딥링크 현지화 | 카카오내비/티맵은 한국 전용 — 구글맵, 현지 내비 앱으로 딥링크 체인 교체 |

### 아키텍처 변경
- **버전1에서 이미 대비되어 있음**: `MapView.tsx`가 카카오맵 SDK 호출을 캡슐화하고 있어, 지도 레이어를 Leaflet+OSM 등으로 교체할 때 이 컴포넌트만 수정하면 나머지(거리계산, 카드 UI, 상세시트)는 그대로 재사용 가능
- 카카오 로컬 API로 확보한 좌표는 카카오 약관상 타 지도 서비스에 영구 사용이 제약될 소지가 있음(버전1 개발계획서 3장 기록) — 해외 확장 시 좌표 재획득(VWorld 운영키 재지오코딩 또는 현지 지오코딩 서비스) 필요
- `src/lib/navi.ts`의 딥링크 체인을 국가별로 분기하는 구조로 재설계 필요

## 우선순위 및 선행 조건 요약

| 버전 | 핵심 선행 조건 | 상대적 개발 난이도 |
|---|---|---|
| 버전2 | 백엔드 도입 결정(가장 큰 분기점) | 높음 |
| 버전3 | 지하철 공공데이터 확보 | 중간 |
| 버전4 | 없음(버전1 구조로 즉시 시작 가능) | 낮음 |
| 버전5 | 지도 SDK 교체, 좌표 재획득, 해외 데이터 확보 | 매우 높음 |

**권장 순서**: 버전1 안정화(실차 테스트, 지오코딩 실패분 구제) → 버전4(다국어, 낮은 난이도로 사용자 저변 확대) → 버전2(수요 검증 후 백엔드 투자 결정) → 버전3 → 버전5.
