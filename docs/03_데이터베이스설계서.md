# 화장실SOS(RestroomMap) 데이터베이스 설계서

| 항목 | 내용 |
|---|---|
| 문서버전 | v1.0 |
| 작성일 | 2026-07-17 |

> 본 시스템은 버전1 원칙(백엔드 없음)에 따라 **RDBMS를 사용하지 않는다.** 대신 정제된 데이터를 정적 JSON 파일로 배포하고, 클라이언트가 필요한 파일만 fetch하는 구조다. 아래는 이 정적 데이터의 **논리 스키마**를 RDB 설계서 형식(컬럼ID/컬럼명/타입/길이/PK/notnull)으로 표현한 것이다.

---

## 1. 데이터 저장소 구조

| 구분 | 저장 형태 | 위치 |
|---|---|---|
| 원본 입력 데이터 | CSV (CP949→UTF-8 변환) | `data/공중화장실정보_utf8.csv` (Git 미포함, 로컬 전용) |
| 지오코딩 중간 산출물 | JSONL (1행 1레코드) | `data/geocoded.jsonl` (Git 미포함) |
| **서비스 데이터(최종)** | **정적 JSON, geohash 5자리 단위 분할** | `public/tiles/{geohash5}.json` × 4,356개 (Git 포함, 실배포) |
| 타일 인덱스 | 단일 JSON | `public/tiles/_index.json` |

## 2. 엔터티: Restroom (화장실) — 서비스 데이터 논리 스키마

타일 JSON 배열의 각 원소가 1개 레코드에 해당한다. TypeScript 타입 정의: `src/types.ts`의 `Restroom` 인터페이스.

| 컬럼ID | 컬럼명 | 타입 | 길이 | PK | NOT NULL | 설명 |
|---|---|---|---|---|---|---|
| 01 | id | VARCHAR | 18 | **PK** | Y | 원본 관리번호(공공데이터 "관리번호" 그대로 사용) |
| 02 | name | VARCHAR | 100 |  | Y | 화장실명 |
| 03 | lat | DECIMAL | (9,6) |  | Y | 위도(WGS84), 소수 6자리(≈11cm 정밀도)로 반올림 |
| 04 | lng | DECIMAL | (9,6) |  | Y | 경도(WGS84) |
| 05 | roadAddr | VARCHAR | 200 |  | N | 도로명주소 |
| 06 | jibunAddr | VARCHAR | 200 |  | N | 지번주소 |
| 07 | openHours | VARCHAR | 200 |  | N | 개방시간 원문(자유서식 텍스트, 상세 우선 표시) |
| 08 | maleToilets | INT | - |  | N | 남성용 변기수(대+소변기 합산) |
| 09 | femaleToilets | INT | - |  | N | 여성용 변기수(대변기) |
| 10 | disabled | BOOLEAN | - |  | N | 장애인용 변기 유무(남녀 장애인용 합산 > 0) |
| 11 | diaperTable | BOOLEAN | - |  | N | 기저귀교환대 유무(불명확 시 필드 자체 생략) |
| 12 | emergencyBell | BOOLEAN | - |  | N | 비상벨 설치 유무 |
| 13 | cctv | BOOLEAN | - |  | N | 화장실입구 CCTV 설치 유무 |
| 14 | managerTel | VARCHAR | 20 |  | N | 관리기관 전화번호 |
| 15 | source | ENUM | - |  | N | `public`\|`user` — **버전2 예약 필드**(현재 전부 `public`) |
| 16 | category | ENUM | - |  | N | `public`\|`subway`\|`private` — **버전3 예약 필드**(현재 전부 `public`) |

> **NOT NULL 판단 기준**: 01~04는 지오코딩·정제 파이프라인에서 결측 시 해당 레코드 자체를 제외하므로 항상 존재. 05~16은 원본 데이터 결측 시 `undefined` 처리 후 JSON 직렬화 단계에서 키 자체를 제거(용량 절감) — 즉 "컬럼 없음"과 "값 없음"을 구분하지 않고 통일.

### 2.1 파생 필드 (런타임 전용, 저장하지 않음)

| 컬럼명 | 타입 | 설명 |
|---|---|---|
| distance | FLOAT | 클라이언트가 현재 위치 기준 Haversine 공식으로 실시간 계산(m 단위). `RestroomWithDistance` 타입에서만 존재 |

## 3. 엔터티: TileIndex (타일 인덱스)

`public/tiles/_index.json` — 단일 파일, 배열이 아닌 객체.

| 컬럼ID | 컬럼명 | 타입 | 길이 | PK | NOT NULL | 설명 |
|---|---|---|---|---|---|---|
| 01 | precision | INT | - |  | Y | geohash 정밀도(자릿수), 고정값 5 |
| 02 | total | INT | - |  | Y | 전체 레코드 수(50,591) |
| 03 | tiles | MAP<VARCHAR(5), INT> | - | **PK(키)** | Y | 타일 키(geohash5) → 해당 타일 내 레코드 수 |

## 4. 엔터티: DataMeta (데이터 생성 메타)

`src/data-meta.json` — 빌드 시점에 앱 번들에 포함(런타임 fetch 아님).

| 컬럼ID | 컬럼명 | 타입 | 길이 | PK | NOT NULL | 설명 |
|---|---|---|---|---|---|---|
| 01 | updatedAt | VARCHAR | 10 |  | Y | 타일 생성 시각, `YYYY.MM.DD` 형식. `npm run tiles` 실행 시마다 자동 갱신 |

`data/공중화장실정보_utf8.csv` — 총 34개 컬럼, 53,543행. 서비스 스키마(2장)로 정제되기 전 원본이며, Restroom 엔터티 제작에 사용된 컬럼만 발췌한다.

| 컬럼ID | 원본 컬럼명 | 매핑 대상(Restroom) | 비고 |
|---|---|---|---|
| C01 | 관리번호 | id | 유일키로 사용 |
| C02 | 화장실명 | name | |
| C03 | 소재지도로명주소 | roadAddr | 지오코딩 1차 입력 |
| C04 | 소재지지번주소 | jibunAddr | 지오코딩 2차 입력(도로명 실패 시) |
| C05 | 남성용-대변기수 | maleToilets(합산) | |
| C06 | 남성용-소변기수 | maleToilets(합산) | |
| C07 | 남성용-장애인용대변기수 | disabled(합산 후 boolean화) | |
| C08 | 남성용-장애인용소변기수 | disabled(합산 후 boolean화) | |
| C09 | 여성용-대변기수 | femaleToilets | |
| C10 | 여성용-장애인용대변기수 | disabled(합산 후 boolean화) | |
| C11 | 관리기관명 | (미사용, managerTel만 채택) | |
| C12 | 전화번호 | managerTel | |

| C15 | 비상벨설치여부 | emergencyBell | |
| C16 | 화장실입구CCTV설치유무 | cctv | |
| C17 | 기저귀교환대유무 | diaperTable | |

> 원본 CSV의 나머지 컬럼(구분명, 근거법령명, 어린이용 변기수, 설치연월, 오물처리방식, 리모델링연월, 데이터기준일자 등)은 현재 버전1 서비스 화면에서 사용하지 않아 매핑하지 않았다. 향후 필요 시 `scripts/build-tiles.mjs`의 `IX` 매핑 객체에 컬럼을 추가하면 된다.

### 5.1 원본 데이터 품질 이슈
- 위경도 컬럼 없음(2025-02부터 정책 중단) → 지오코딩 필수
- 도로명/지번 주소 뒤바뀜, 주소 꼬리말(예: "평동 210 서대문역") 존재 → `geocode.mjs`에서 정제
- 34개 컬럼 중 위경도가 없다는 점을 제외하면 전반적으로 결측률은 낮은 편(개방시간상세만 약 40%가 공란)

## 6. 데이터 정합성 규칙 (빌드 타임 검증)

| 규칙 | 처리 |
|---|---|
| 좌표 결측 | 해당 레코드 제외 (1,269건, 2.4%) |
| 좌표 범위 이탈(한국 영역: 위도 33~39, 경도 124~132 밖) | 해당 레코드 제외 |
| 중복 레코드(좌표 소수 4자리 + 명칭 동일) | 제외 (1,683건) |
| 시설 유무 텍스트 → boolean 변환 | "미설치/없음/불가/N/무" → false, "설치/있음/가능/Y/유/O" → true, 그 외 → 필드 생략(unknown) |

## 7. ER 다이어그램 (개념)

```
┌─────────────────┐        1:N        ┌──────────────────┐
│   TileIndex      │ ─────────────────▶│   TileFile         │
│  (_index.json)   │  tiles[geohash]   │ ({geohash5}.json)  │
└─────────────────┘                    └────────┬──────────┘
                                                 │ 1:N (배열)
                                                 ▼
                                        ┌──────────────────┐
                                        │    Restroom       │
                                        │ (id가 논리적 PK)   │
                                        └──────────────────┘
```

관계형 DB의 외래키 제약은 없으며(정적 파일이므로), geohash 값 자체가 TileIndex와 실제 파일명을 연결하는 암묵적 키 역할을 한다.
