## 1. 최초 요구사항

건국대학교 공학대학원 AI 전공 겸임교수 양희정 님이 개인 사이드 프로젝트로 제안:

> "운전 중 급할 때 가장 가까운 공공화장실을 3초 안에 찾고, 터치 2번 이내로 내비 안내까지 연결되는 모바일 웹앱을 만들고 싶다. 백엔드 없이 0원으로 운영 가능해야 한다."

핵심 제약:
- 백엔드 없음(정적 호스팅만)
- 모바일 우선, 큰 버튼·고대비(운전 중 사용 고려)
- 위치정보 기기 밖 미전송(프라이버시)
- 향후 버전2(민간 제보)~버전5(해외) 확장을 염두에 둔 스키마 설계

## 2. 진행 경과 (시간순)

### 2.1 Cowork 세션 (데스크톱 채팅) — 데이터 파이프라인 + Phase 0/2/3
카카오/VWorld API 등 외부 인터넷이 샌드박스에서 차단되는 환경 제약으로, 지오코딩·타일 생성·개발서버 실행은 사용자 PC에서 직접 수행:
- 공공데이터포털 CSV(53,543건, CP949) 확보
- **핵심 이슈 발견**: 표준데이터의 위경도 제공이 2025년 2월부터 정책적으로 중단됨 → 카카오 로컬 API로 자체 지오코딩 파이프라인 구축, 52,274건(97.6%) 성공
- geohash5 타일 분할(4,356개), 중복 1,683건 제거
- Vite+React+TS+PWA 스캐폴딩(Phase 0), 지도+현재위치(Phase 2), 주변검색+리스트(Phase 3) 완료

### 2.2 Claude Code 세션 인수 — Phase 4~6
인수인계 문서(`진행상황_인수인계.md`)를 읽고 Phase 4부터 이어서 진행. Claude Code는 실제 PC 환경에서 직접 실행되어 이전 세션의 샌드박스 제약이 없었음.

**Phase 4 — 상세 바텀시트 + 내비 딥링크**
- 마커/카드 탭 → 상세정보(개방시간, 변기수, 장애인용, 기저귀교환대, 비상벨, CCTV, 관리기관 전화) 바텀시트
- 안내 시작: 카카오내비 → 티맵 → 카카오맵 웹 순 폴백 체인(`visibilitychange` 이벤트로 앱 전환 감지)
- **시행착오**: `VITE_KAKAO_JS_KEY`를 넣었는데도 지도가 안 뜸 → 카카오 개발자센터에 `localhost:5173` 도메인이 미등록된 게 원인(401 domain mismatched)이었음. 이후 배포 도메인 변경 때마다 동일한 이슈가 반복되어, 매번 "도메인 등록 → curl로 401/200 확인" 절차를 정형화함

**Phase 5 — PWA·오프라인·다크모드**
- 라이트/다크 CSS 변수 체계 도입, 기기 다크모드 설정 자동 반영
- Service Worker 타일 캐싱(StaleWhileRevalidate) 구조 확인

**Phase 6 — 배포**
- 기존 `C:\AI` 저장소는 무관한 교육사이트 프로젝트였기 때문에, RestroomMap 전용 GitHub 저장소(`YangGuiBee/RestroomMap`)를 신규 생성해 분리
- Vercel(Hobby) 연동, GitHub push 시 자동 재배포 확인
- 카카오 개발자센터에 배포 도메인 추가 등록

### 2.3 기능 고도화 (사용자 피드백 반복 사이클)
- 특정위치 검색(카카오 Places API 연동) — 지금 위치가 아닌 목적지 주변 사전 검색
- 지도 마커에 "화" 고정 문자 대신 거리순 순위 숫자 표시
- 하단 카드에 순위 배지(1위 강조) + 페이지네이션(‹‹/››)
- 카드 개수 반응형화: 모바일 3개 / 데스크톱 6개(`useIsDesktop` 훅, `min-width:768px` 기준)
- 개방시간 텍스트에 요일 정보가 있는데도 "정시/상시" 같은 분류값에 가려 안 보이던 **데이터 우선순위 버그**를 발견해 수정
- `src/lib/openHours.ts` 신규 개발: 자유서식 개방시간 텍스트를 정규식으로 파싱해 "지금 이용 가능한지" 판정. 애매하면 표시를 생략하는 fail-safe 원칙 적용(오탐이 실제 열린 곳을 놓치게 하는 것보다 위험하다는 판단)

### 2.4 지도 마커 렌더링 디버깅 — 가장 긴 시행착오
사용자가 "지도에 특정 번호 마커가 안 보인다"고 반복 제보. 원인이 매번 달라 총 5단계에 걸쳐 원인을 좁혀감.

| 단계 | 발견한 원인 | 조치 |
|---|---|---|
| 1 | 같은 건물의 서로 다른 화장실이 지오코딩 좌표가 완전히 동일 → 나중 마커가 앞 마커를 완전히 가림 | 골든 앵글(137.5°) 오프셋으로 좌표 자체를 살짝 벌림 |
| 2 | **더 근본적 문제**: 카카오맵 CustomOverlay는 뷰포트 밖 좌표를 아예 DOM에 그리지 않는데, 지도 줌 레벨을 고정값(4)으로 써서 먼 마커가 화면 밖으로 밀려남 | `LatLngBounds` + `setBounds()`로 표시할 마커 전체가 보이도록 자동 확대/축소, 과도한 축소 방지 상한(레벨6) 병행 |
| 3 | 마커가 멀리 퍼져 있으면 자동 축소가 너무 과해져, 오히려 가까운 마커끼리 화면 픽셀상 겹침(위경도 오프셋만으로는 줌 레벨마다 필요 간격이 달라 무의미) | 카카오맵 Projection API(`pointFromCoords`/`coordsFromPoint`)로 실제 화면 픽셀 좌표를 계산해, 링(반경) 단위로 점점 넓혀가며 충돌 없는 자리를 탐색하는 방식으로 교체 |
| 4 | 사용자 UX 피드백: "근처 화장실 찾기가 목적인데 처음부터 다 보이게 넓힐 필요 없다" | 지도가 **항상 현재 카드 페이지(모바일 3/데스크톱 6개)만** 기준으로 맞춰지도록 설계 변경. `>>` 페이지 전환 시 그 페이지 기준으로 재조정. 부수 효과로 픽셀 겹침도 크게 줄어듦 |
| 5 | 그럼에도 실제 사용자 브라우저에서만 재현되는 미세한 겹침 잔존(Claude 테스트 환경에서는 재현 안 됨 — 브라우저 확대율/모니터 배율 차이로 추정) | 최소 픽셀 간격을 26px → 36px → **50px**까지 단계적으로 확대해 최종 해결 확인 |

**교훈**: 지도 마커 렌더링은 "좌표가 맞다"와 "화면에 실제로 보인다"가 별개 문제다. SDK의 뷰포트 클리핑, 줌 레벨에 따른 상대 픽셀 거리 변화, 브라우저별 서브픽셀 렌더링 차이까지 고려해야 완전히 해결된다.

### 2.5 브랜딩 및 산출물 정리
- 앱 이름을 "화장실맵" → **"화장실SOS"**로 변경(브랜드명만 변경, 저장소/도메인 등 기술 식별자는 `RestroomMap` 유지 — 재작업 비용 절감)
- 배포 도메인을 `restroommap.vercel.app` → **`wc-sos.vercel.app`**로 변경(희망했던 `wcsos.vercel.app`은 타 팀이 선점해 실패, 대안으로 확정)
- 하단 안전 문구를 저작권 표기로 교체
- 현대자동차/기아 오픈이노베이션 제안서 작성(비공개, 로컬 전용 문서)
- 버전1 마무리 산출물(설계서·매뉴얼·개발일지·향후계획) 정리

## 3. 주요 의사결정과 그 이유

| 결정 | 이유 |
|---|---|
| 백엔드 없이 정적 호스팅만 사용 | 0원 운영, 유지보수 부담 최소화(KISS/YAGNI) |
| 관리자 대시보드(로그인+데이터 재수집 UI) 요청을 버전2로 이연 | 백엔드·인증 시스템이 필요해 버전1 원칙과 정면 충돌. GitHub Actions 수동 트리거 등 대안도 검토했으나 "정식 관리자 화면"은 아니라 판단해 보류 |
| 지도 SDK를 카카오맵으로 고정하되 레이어 추상화 유지 | 국내 정확도·카카오내비 연계성 우선, 향후 해외 확장 시 OSM 전환 여지 확보 |
| 좌표 오프셋보다 픽셀 기반(Projection API) 충돌 회피로 전환 | 위경도 오프셋은 줌 레벨마다 필요 크기가 달라 근본 해결이 안 됨을 실측으로 확인 |
| 앱 브랜드명과 기술 식별자(저장소명/도메인) 분리 | 이름 변경 시마다 도메인 재등록·재배포 카스케이드를 피하기 위함(오늘 실제로 도메인 변경 작업의 번거로움을 겪은 뒤 내린 결정) |
