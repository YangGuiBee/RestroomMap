## 1. 이관 개요

버전1(공공화장실)이 실배포 완료된 상태에서, 운영과 버전2~5 개발을 신규 담당자에게 넘긴다. 신규 담당자는 IT 비전공자로 가정하며, 이 문서는 **"무엇을 이관해야 하는지"**를, 별첨 `09_신규담당자_계정설정가이드.md`는 **"신규 담당자가 실제로 어떤 순서로 클릭해야 하는지"**를 다룬다.

### 1.1 이관해야 할 4가지 축
| 축 | 내용 |
|---|---|
| ① 소스코드 | GitHub 저장소 접근 권한 |
| ② 배포 인프라 | Vercel 프로젝트 접근 권한 |
| ③ 외부 API 계정 | 카카오 개발자센터(지도), data.go.kr(원본데이터), VWorld(지오코딩 폴백) |
| ④ 비밀정보(.env) | REST/JS 키 등 — Git에 없으므로 별도 전달 필요 |

## 2. 이관 방식 결정: 계정 자체 이전 vs 협업자 초대

| 방식 | 장점 | 단점 |
|---|---|---|
| A. 계정/프로젝트 소유권 완전 이전 | 원 담당자 완전히 손 뗌 | 플랫폼별 이전 절차가 번거롭고, 서비스 중단 리스크가 있음(특히 카카오 앱은 소유권 이전 기능이 제한적일 수 있음) |
| **B. 협업자/팀원으로 초대 (권장)** | 기존 URL·API 키·도메인 등록이 그대로 유지되어 **서비스 중단 없음**. 신규 담당자가 자기 계정으로 로그인해서 바로 작업 가능 | 원 담당자 계정이 명목상 계속 존재(필요 시 나중에 완전 이전 가능) |

**본 계획서는 방식 B(협업자 초대)를 기준으로 작성한다.** 신규 담당자가 완전히 홀로서기 하려면 이관 완료 후 별도로 각 플랫폼의 "소유권 이전" 절차를 추가로 밟으면 된다(3.5절 참고).

## 3. 단계별 이관 절차

### 3.1 GitHub 저장소
- 저장소: https://github.com/YangGuiBee/RestroomMap
- **작업(원 담당자)**: 저장소 → Settings → Collaborators → 신규 담당자의 GitHub 계정을 **Collaborator**로 추가(또는 Admin 권한 부여)
- **작업(신규 담당자)**: 이메일로 온 초대 수락 → 저장소 접근 가능 확인

### 3.2 Vercel 배포
- 프로젝트: `wcsos` (팀: YangHeeJung, Hobby 플랜) → 배포 URL `https://wc-sos.vercel.app`
- **작업(원 담당자)**: Vercel 팀 설정 → Members → 신규 담당자 이메일로 초대(역할: Member 이상)
- **작업(신규 담당자)**: 초대 수락 → 팀 진입 확인 → `wcsos` 프로젝트에서 Deployments/Environment Variables 열람 가능한지 확인
- 참고: GitHub 저장소에 push하면 Vercel이 자동 재배포하므로, 신규 담당자는 Vercel 팀원이 되는 순간 사실상 배포 권한을 갖게 됨

### 3.3 카카오 개발자센터
- 앱: `restroommap` (ID 1516089)
- **작업(원 담당자)**: 카카오 개발자센터 → 해당 앱 → 앱 설정에서 멤버 초대 기능이 있는지 확인 후 신규 담당자의 카카오 계정을 멤버로 추가
  - ⚠️ 카카오 개발자센터는 멤버 초대 메뉴 위치가 시점에 따라 바뀔 수 있으니, 안 보이면 카카오 개발자센터 고객센터에 문의
- **대안**: 멤버 초대가 여의치 않으면, 아래 키 값을 **보안 채널로(이메일 평문 금지, 비밀번호 관리자 공유 등 활용)** 신규 담당자에게 직접 전달하고, 카카오 앱 자체는 원 담당자 계정에 당분간 유지
  - REST API 키, JavaScript 키(둘 다 `.env`에 있음)
- 도메인 등록(`localhost:5173`, `https://wc-sos.vercel.app`, `https://restroommap.vercel.app`)은 앱에 이미 되어 있으므로 **키를 그대로 쓰는 한 재등록 불필요**

### 3.4 .env 비밀정보 전달
`.env`는 Git에 포함되지 않으므로 위 계정 이관과 별개로 반드시 직접 전달해야 한다.

| 키 | 용도 | 비고 |
|---|---|---|
| `KAKAO_REST_KEY` | 지오코딩(주소→좌표) | 카카오 개발자센터에서도 재확인 가능 |
| `VWORLD_KEY` | 지오코딩 폴백 | **만료일 2027-01-17** — 이관 후 신규 담당자가 본인 명의로 재발급받는 것을 권장 |
| `VITE_KAKAO_JS_KEY` | 지도 표시(앱 배포용) | Vercel 프로젝트 Environment Variables에도 이미 등록되어 있음 |

전달 방법: 비밀번호 관리자(1Password 등) 공유, 또는 대면/사내 메신저의 암호화 전달 등 보안 채널 사용. **이메일 평문 전송은 지양.**

### 3.5 (선택) 완전한 소유권 이전
신규 담당자가 완전히 독립하길 원하면, 3.1~3.3의 협업자 관계를 아래처럼 승격:
- GitHub: Settings → Transfer ownership
- Vercel: 프로젝트를 신규 담당자의 팀/계정으로 Transfer(Vercel 대시보드 Project Settings 내 Transfer 메뉴)
- 카카오 개발자센터: 앱 소유자 변경 절차(지원 여부는 카카오 정책에 따름 — 안 되면 신규 앱을 새로 만들고 키·도메인을 재등록하는 방법으로 대체)

## 4. 소스코드·데이터 파일 전달 (Git으로 자동 전달되는 것 vs 별도 전달)

**"C:\AI\RestroomMap 폴더 전체 복사"는 권장하지 않는다.** 대신 GitHub 저장소를 `git clone`받는 방식을 기본으로 하고, 아래 "별도 전달" 항목만 추가로 넘긴다.

### 4.2 전달 방법 요약

| 구분 | 항목 | 전달 방법 |
|---|---|---|
| **Git으로 자동 전달됨** | `src/`, `scripts/`, `docs/`, `public/`(타일 4,357개 포함), 설정파일, 루트의 이미지·기획 문서 일체 | 신규 담당자가 `git clone`만 하면 전부 받아짐 |
| **별도 전달 필요 — 필수** | `.env` (API 키 3종) | 3.4절 방식으로 보안 전달 |
| **별도 전달 필요 — 권장(시간 절약)** | `data/` 폴더 전체(공중화장실정보_utf8.csv 19MB, geocoded.jsonl 5.5MB, geocode_failed.jsonl 180KB), 루트의 `공중화장실정보.csv`(16MB, 원본) | 압축 후 파일 전송(사내 드라이브 등). 없어도 되지만, 없으면 지오코딩(카카오 API 일 10만건 쿼터, 약 반나절~하루 소요)을 처음부터 다시 돌려야 함 |
| **불필요 (자동 생성됨)** | `node_modules/`, `dist/` | 각각 `npm install`, `npm run build`로 로컬에서 재생성 |
| **선택(대외비 — 원 담당자 판단)** | `제안서_현대기아_오픈이노베이션.md` | Git에서 의도적으로 제외(`.gitignore`)된 대외 제안서. 신규 담당자에게 넘길지는 원 담당자가 별도 판단 |

> **결론**: 신규 담당자는 (1) GitHub 저장소를 clone하고 (2) `.env` 파일을 별도로 전달받아 프로젝트 루트에 생성하면, 코드·이미지·기획 문서 관점에서는 이관이 끝난다. `data/` 폴더와 루트의 원본 CSV는 있으면 편하지만 없어도 `npm run geocode`부터 다시 돌리면 재생산 가능하다.

## 5. 이관 후 확인 체크리스트

- [ ] 신규 담당자가 GitHub 저장소에 접근되는지 (Collaborator 초대 수락 완료)
- [ ] 신규 담당자가 Vercel 팀에 접근되는지 (팀 초대 수락 완료)
- [ ] 신규 담당자가 로컬에서 `git clone` → `.env` 생성 → `npm install` → `npm run dev` 실행 후 `localhost:5173`에서 지도가 정상 표시되는지
- [ ] 신규 담당자가 임의로 코드 한 줄 수정 → `git push` → Vercel 자동 재배포 → `https://wc-sos.vercel.app`에 반영되는지 (엔드투엔드 배포 파이프라인 검증)
- [ ] 신규 담당자가 카카오 개발자센터에서 앱 키 위치를 확인할 수 있는지(멤버 초대 or 키 전달 중 택1 완료)
- [ ] `data/` 원본 파일 전달 여부 결정 및 완료
